# Dashboard UI

## Stack: Vue 3 + Vite + TypeScript + TailwindCSS + ECharts

---

## Overview

The dashboard is a real-time ML operations platform with 5 pages, all communicating with the FastAPI backend via REST APIs and WebSockets.

---

## Technology Stack

| Layer | Technology | Purpose |
|-------|------------|---------|
| Framework | Vue 3 | Reactive UI |
| Build Tool | Vite | Fast development/build |
| Language | TypeScript | Type safety |
| Styling | TailwindCSS | Dashboard UI |
| Charts | Apache ECharts | Real-time telemetry charts |
| Routing | Vue Router | Multi-page dashboard |
| State | Pinia | Lightweight state management |
| Realtime | WebSockets + SSE | Live telemetry + log streaming |
| HTTP Client | Axios | API communication |

## Pages

### PAGE 1 — Fleet Command Center (`/`)

Live fleet overview powered by `/ws/predictions` and `/ws/alerts` WebSocket streams.

Components:
- 4 stat cards: Active Engines, Critical, High Risk, Avg Fleet RUL
- Risk distribution pie chart
- RUL bar chart (top engines by risk)
- Engine status table (sortable, color-coded risk, click → engine detail)
- Alerts panel (HIGH + CRITICAL engines)

---

### PAGE 2 — Engine Detail (`/engine/:id`)

Per-engine deep dive.

Components:
- Risk gauge, RUL, confidence, prediction timestamp
- Sensor tag display
- Engine metadata (source, buffer size, last prediction)

---

### PAGE 3 — Streaming Pipeline Monitor (`/pipeline`)

Live pipeline and infrastructure status.

Components:
- 4 live counters: Active Engines, Telemetry Engines (Redis), Avg Cycle, Critical — all from `engineStore`
- Live telemetry feed table (engine ID, cycle, window size, last event time)
- Service health checks with green/red/checking indicators (FastAPI, Redis, Solace, Flink, Prometheus, Grafana) — polled every 30s
- Pipeline topology diagram with colour-coded stage cards
- Quick start commands

---

### PAGE 4 — ML Observability (`/mlops`)

Model registry, metrics, drift monitoring, and retraining.

Components:
- **Retrain Model button** — triggers `POST /pipeline/run`, disabled while running
- **Pipeline status indicator** — Idle / ⟳ Running… / ✓ Completed / ✗ Failed
- **Live log panel** — SSE stream from `GET /pipeline/logs`, color-coded by log level, auto-scrolls
- Active model info panel (type, version, input shape, RUL clip, sensors)
- GRU architecture diagram
- Performance metrics table (RMSE, NASA Score, Precision/Recall/F1 Critical, Accuracy, F1 Weighted) — auto-refreshes after successful retraining
- Drift monitoring links (Prometheus, Grafana)
- **Evidently reports list** — fetched from `GET /drift/reports`, click any row → full-screen iframe modal with interactive Evidently dashboard

---

### PAGE 5 — Replay & Simulation Lab (`/replay`)

Synthetic telemetry simulation for testing and demos.

Components:
- Engine ID input + speed selector (×1/×5/×10/×20 events/sec)
- Start/Stop/Reset controls
- 4 failure injection modes:
  - None
  - 🔥 Overheating — spikes temperature sensors s3, s4 by +25%
  - 📉 Pressure Drop — drops pressure sensors s9, s11 by −30%
  - 📳 Vibration Spike — spikes s14, s17 by +35% with high variance
- 4 stat cards: Cycles Pushed, Current RUL, Failure Risk, Risk Level
- Live prediction feed table (last 50 predictions, color-coded by risk)

**How it works:**
1. Each tick calls `POST /push` → writes to `engine:{id}:buffer` (Redis list)
2. After 30 cycles the buffer is full
3. Page polls `GET /predict/stream/{id}` every 2s → reads buffer → GRU → prediction
4. Failure injection shifts sensor values outside training distribution → model predicts higher risk

## State Management (Pinia)

| Store | File | Purpose |
|-------|------|---------|
| `engineStore` | `stores/engineStore.ts` | Predictions map, telemetry map, model info, WS connection state |
| `alertStore` | `stores/alertStore.ts` | Alert list, acknowledgement state |

`engineStore` is the single source of truth for all live data — populated by the `useWebSockets` composable which connects all three WS streams on mount.

---

## WebSocket Architecture

Three streams, all managed by `composables/useWebSockets.ts`:

| Stream | Endpoint | Interval | Payload |
|--------|----------|----------|---------|
| Predictions | `/ws/predictions` | 5s | Batched GRU inference on all Redis engines |
| Telemetry | `/ws/telemetry` | 2s | Engine metadata (cycle, event_time, window_size) |
| Alerts | `/ws/alerts` | 5s | HIGH + CRITICAL engines only |

The WS streams only cover engines in the **Redis feature store** (written by the streaming consumer). The Replay page uses polling instead because it writes to the push buffer, not the feature store.

## API Services (`services/api.ts`)

```typescript
getHealth()                    // GET /health
getModelInfo()                 // GET /model/info
getModelEvaluation()           // GET /model/evaluation
getEngines()                   // GET /engines
getEngine(id)                  // GET /engines/{id}
getAlerts(minRisk)             // GET /alerts
predictEngine(id)              // GET /predict/engine/{id}
getDriftReports()              // GET /drift/reports
triggerPipeline()              // POST /pipeline/run
getPipelineStatus()            // GET /pipeline/status
```

Pipeline logs are consumed via native `EventSource` (SSE), not Axios.


## Deployment

Built as a static site served by nginx. The nginx config (`nginx.conf`) proxies all API and WebSocket routes to the inference API container:

```nginx
location /ws/          { proxy_pass http://inference-api:8000/ws/; ... }
location ~ ^/(health|model|predict|push|engines|alerts|metrics|drift|pipeline) {
    proxy_pass http://inference-api:8000;
}
location / { try_files $uri $uri/ /index.html; }
```
